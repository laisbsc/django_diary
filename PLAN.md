# django_diary — Development Plan

## Goal
Expand django_diary into a production travel blog with AI-assisted post creation, deployed on Render.

---

## Phase 1 — Render Migration ✅ DONE

- Removed Heroku (`django-on-heroku`, `Procfile`, `requirements.txt`)
- Split settings into `travel_diaries/settings/base.py`, `local.py`, `production.py`
- Added `gunicorn`, `whitenoise[brotli]`, `django-storages[s3]`, `psycopg[binary,pool]`, `dj-database-url`
- Created `render.yaml` (IaC): 2 web services (prod + staging) + 2 PostgreSQL DBs
- Package manager: `uv` with `pyproject.toml` + `uv.lock`
- Created `staging` branch (deploys to staging service), `main` deploys to prod
- Created `.env.example`

### Render infrastructure
| Service | Branch | Region | Plan |
|---|---|---|---|
| django-diary | main | frankfurt | starter |
| django-diary-staging | staging | frankfurt | starter |
| django-diary-db | — | frankfurt | starter ($7/mo) |
| django-diary-db-staging | — | frankfurt | free (ephemeral) |

### Manual env vars to set in Render dashboard (both services)
- `ALLOWED_HOSTS` — comma-separated domains
- `CSRF_TRUSTED_ORIGINS` — `https://yourdomain.com,...`
- `LOGFIRE_TOKEN`
- `CLOUDFLARE_R2_BUCKET_NAME`
- `CLOUDFLARE_R2_ACCESS_KEY_ID`
- `CLOUDFLARE_R2_SECRET_ACCESS_KEY`
- `CLOUDFLARE_R2_ENDPOINT_URL` — `https://<account-id>.r2.cloudflarestorage.com`

`SECRET_KEY` is auto-generated by Render (`generateValue: true`). It is a plain string —
the base64 decode in `base.py` will break for Render-generated keys. **This needs resolving
before first deploy**: either base64-encode the Render key manually before setting it, or
add a fallback in `base.py` that tries plain string if base64 decode fails.

### Cloudflare R2
- Single bucket, folder prefix per environment: `prod/` and `staging/`
- Media files only (post images, uploads)
- Static files served by whitenoise

### DNS
- Domain registered on Namecheap
- Point to Render after services are created (Render provides CNAME/A record)

### Local dev
```
uv run python manage.py runserver
```
Uses `settings/local.py` (SQLite, DEBUG=True) automatically.

### Build command (run by Render on each deploy)
```
pip install uv && uv sync --frozen && uv run python manage.py collectstatic --no-input && uv run python manage.py migrate
```

### Start command
```
uv run gunicorn travel_diaries.wsgi:application --bind 0.0.0.0:$PORT --workers 2
```

## Phase 4 — AI Post Creation (Pydantic AI)

### Goal
Take raw travel notes (from Obsidian or Google Notes) and generate a draft blog post.

### Questions still open — answer before starting this phase
1. **Source**: Obsidian or Google Notes? Which do you use day-to-day?
2. **Input method**: manually paste notes into a form, or auto-pull via API/sync?
3. **Output**: full draft post ready to review, or structured outline only?
4. **Location enrichment**: should AI auto-populate the `Location` model from notes?
5. **Trigger**: Django view (e.g. `/posts/from-notes`), Django admin page, or CLI script?
6. **Edit before publish**: yes (review page) or one-shot generate-and-publish?
7. **Pydantic AI experience**: first time or familiar?
8. **LLM**: Claude (recommended), OpenAI, or other?

### Likely architecture (to be confirmed after answers above)
- New Django app: `ai/` or feature added to `blog/`
- Pydantic AI agent with structured output matching the `Post` model schema
- Django view with a textarea (paste notes) → POST → agent → draft post review page → publish
- `ANTHROPIC_API_KEY` (or other) added to env vars and `render.yaml`
- Logfire already configured — AI calls will be traced automatical1ly
